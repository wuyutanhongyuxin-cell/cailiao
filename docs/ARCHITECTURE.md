# 架构与实施方案

## 核心判断

材料写作的主要风险不是“文风不够像”，而是事实缺失、引用失真、执行要素不全，以及模型在自检时放过自己的错误。因此系统分为四层：

| 层次 | 职责 | 是否依赖大模型 |
|---|---|---|
| 事实层 | 保存任务事实、证据、来源和有效性 | 否 |
| 规则层 | 检查必填项、结构、数字、政策引用和执行要素 | 否 |
| 生成层 | 根据已批准事实组织语言 | 是，可替换 |
| 验证层 | 检查结果并决定是否允许导出 | 尽量不依赖；语义验证可用独立模型 |

## 当前结构

```text
浏览器工作台
  ├─ 任务与文种字段
  ├─ 证据台账
  ├─ 审稿问题列表
  └─ 草稿与导出
          │ JSON/HTTP
          ▼
Python 本地服务
  ├─ 文种规则引擎
  ├─ 事实与证据检查
  ├─ OpenAI 兼容客户端
  ├─ SQLite / FTS5
  └─ DOCX 生成
```

当前采用 Python 标准库，使第一版无需安装依赖。后续加入文档解析、语义检索和正式 DOCX 排版后，应使用锁定依赖的工程化方案。

## 目标数据模型

- `Project`：一次材料任务或长期专题
- `Document`：完整政策、文件或内部资料
- `EvidenceChunk`：可检索、可定位的证据片段
- `Fact`：用户确认的事实、数据、时间、主体和结果
- `Claim`：草稿中的可核验主张
- `Citation`：主张与证据的绑定关系
- `DraftVersion`：大纲、段落、锁定状态和版本
- `ReviewIssue`：规则、严重度、定位、修复状态
- `ExportArtifact`：导出文件、模板和校验结果

关键约束：`Claim` 不能因语言合理就被当作事实；它必须绑定 `Fact` 或 `Citation`，否则只能标为待核实。

## 检索方案

1. BM25 处理政策名称、文号、专名和精确表述；
2. 向量检索处理同义表达和问题式查询；
3. RRF 融合两路结果；
4. 元数据过滤地区、机构、时间与有效性；
5. 重排器排序候选片段；
6. 引用验证器确认片段真正支撑主张。

输出必须保留原文、来源、定位、发布日期、抓取时间和有效性，不能只给模型摘要。

## 写作状态机

```text
DRAFT_INTAKE
  ├─ 缺事实 → NEEDS_FACTS
  └─ 通过 → READY_TO_DRAFT → DRAFTED → REVIEWED
                                      ├─ 失败 → NEEDS_REPAIR → REVIEWED
                                      └─ 通过 → READY_TO_EXPORT → EXPORTED
```

每次转换都由明确条件触发，模型不能自行把 `NEEDS_FACTS` 改成 `READY_TO_DRAFT`。

## 模型策略

第一阶段不训练专用小模型。先用可替换的大模型配合证据约束和规则门禁，积累任务、确认事实、证据、初稿、审稿问题、人工修改、最终稿、修改原因和评分。只有评测证明某类稳定、重复、可标注的表达任务仍是瓶颈，且样本充足时，再考虑监督微调。事实正确性和政策时效性始终由检索与验证承担。

## 安全边界

- 默认仅绑定 `127.0.0.1`；
- Key 只从进程环境变量读取；
- 运行数据、导出文件、缓存和环境文件不提交；
- 外部模型调用应明确展示发送的数据范围；
- 本地模型保持同一生成接口；
- 正式材料保留人工审核和签发环节。

## 第三方复用

`MackDing/zh-policy-rag` 使用 MIT 许可证，可在保留许可证与版权声明的前提下评估复用其检索和引用模块。
